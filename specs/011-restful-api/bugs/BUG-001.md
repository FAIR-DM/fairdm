# Bug Report: API URL Name Collision Overrides Portal View Reversals

**Type**: Spec gap
**Severity**: High
**Feature**: 011-restful-api
**Reported**: 2026-04-13
**Status**: Patched
**Patched**: 2026-04-13

## Description

After Feature 011, API endpoints introduced URL names that collide with existing server-rendered portal view names (for example, `project-list` and `dataset-list`).

Because both URL sets are in the global resolver namespace, `reverse()`/`view_name` lookups for common names now resolve to API routes in some contexts instead of the intended template views. This causes portal navigation links (including main menu items) to route users to DRF browsable/API pages rather than the normal list/detail pages.

Observed examples:

- Portal menu item "Projects" (`view_name="project-list"`) can resolve to API list route.
- Portal menu item "Datasets" (`view_name="dataset-list"`) can resolve to API list route.

Likely fix direction: namespace API URLs (for example `api:project-list`) so existing web view names remain unambiguous.

## Artifact Traceability

### spec.md

- **Affected user story**: Story 1 - Read-Only Browsing of Core Data via API (Priority: P1)
- **Also affected**: Story 7 - API Menu Group in Portal Sidebar (Priority: P7)
- **Affected requirements**:
  - FR-001 (auto-generate core model endpoints)
  - FR-002 (auto-generate registered sample/measurement endpoints)
  - FR-017 (menu links use `view_name` for internal API routes)
- **Gap identified**:
  - No requirement states that API URL names must be isolated from existing non-API URL names.
  - No requirement enforces URL naming uniqueness or API namespacing strategy.
  - Spec requires `view_name` usage for internal API links but does not guard against cross-app name collisions.

### plan.md

- **Affected sections**:
  - Phase 3 (US1): router registration and URL wiring for API endpoints.
  - Source code scope entries for `fairdm/api/router.py`, `fairdm/api/urls.py`, and `fairdm/conf/urls.py`.
  - Phase 12 (US7): menu integration relying on URL reversal.
- **Impact**:
  - Plan does not include a namespacing strategy (e.g., `app_name = "api"` plus `include((..., "api"), namespace="api")`).
  - Plan does not include compatibility checks proving portal `project-list`/`dataset-list` reverse results remain mapped to web views after API integration.

### tasks.md

- **Affected tasks**:
  - T017: Registers API core routes with basenames producing `project-list` / `dataset-list` style names.
  - T018: Creates API URL config without explicit URL namespace boundaries.
  - T019: Includes API URLconf into global URL resolver without namespace isolation.
  - T020/T021/T026/T067: Tests validate API behavior/menu links but do not verify no collisions with existing web view names.
- **False completions**:
  - None strictly false by current task wording; tasks were completed as specified but missed a critical integration risk.
- **Missing tasks**:
  - Add API URL namespacing in URLconf and root include.
  - Update internal API menu links to namespaced view names (`api:api-docs`, `api:api-root`).
  - Add regression tests asserting `reverse("project-list")` and `reverse("dataset-list")` resolve to web UI routes, and API routes resolve via `reverse("api:project-list")`/`reverse("api:dataset-list")`.
  - Add migration/compatibility note for any portal code currently reversing unnamespaced API route names.

## Root Cause Analysis

The feature introduced API endpoints into the same global URL name namespace as existing template views, but spec/plan/tasks did not require a collision-avoidance mechanism. As a result, valid API route creation caused unintended resolver ambiguity and navigation regressions in the portal UI.

1. Run `/speckit.bugfix.patch` to update spec artifacts with mandatory API URL namespacing and reverse-lookup compatibility requirements.
2. Run `/speckit.bugfix.verify` to confirm spec, plan, and tasks are consistent after patching.
