# Bug Report: Delete View Has No Name-Confirmation Guard in the UI

**Type**: Implementation drift  
**Severity**: High  
**Feature**: `013-project-crud-views`  
**Reported**: 2026-05-11  
**Status**: Patched  
**Patched**: 2026-05-11

## Description

The project deletion confirmation page presents the user with a plain "Delete" button and no text-input field asking them to type the project name. The name-confirmation check exists only in the server-side `post()` handler â€” if a user clicks the Delete button without any `confirm_name` POST value, the empty string is compared against the project name, so the check passes vacuously for projects whose name is `""` (edge case), and more critically the template gives the user **no affordance** to enter a name at all. In practice, `confirm_name` is absent from the POST body, the comparison `"" != project.name` causes the server to re-render the page with `name_error=True`, but the template does not include a `<input name="confirm_name">` field for the user to fill in, so the user can never satisfy the check and **can also never successfully delete the project through the UI**.

The real risk is the inverse: the template could at any point be connected to `MVPDeleteView`'s standard form rendering (which omits the confirmation input by default because `require_confirmation = False`), making accidental one-click deletion possible.

**Root cause**: The custom `post()` handler implements the name-check logic ad hoc rather than using `MVPDeleteView`'s built-in `require_confirmation = True` mechanism, which renders `DeleteConfirmForm` in the template and validates the typed name through the standard form flow. The spec requires name confirmation (FR-024) and references the `MVPDeleteView` pattern (FR-023), but the implementation bypasses both.

## Root Cause Analysis

Design decision #6 in plan.md sketched a fully custom `post()` method. This approach:

1. Bypasses `MVPDeleteView`'s `require_confirmation` / `DeleteConfirmForm` mechanism entirely.
2. Results in a template that has no `<input name="confirm_name">` field (because the standard template only renders the confirmation input when `require_confirmation = True` is set on the view class).
3. Passes unit tests because the tests inject `confirm_name` directly into the POST body â€” they never render the template and check that the form field exists.

The spec was clear (FR-024) but the design deviated from the platform's standard pattern. The tests were then written to match the implementation rather than the spec requirement.
